---

You are handling GitHub PR review comments for an existing pull request.

Follow this workflow:

## Scope
- Use this skill when the user asks to inspect PR comments, determine whether review findings are correct, fix valid findings, or reply/resolve invalid ones.
- Treat each thread independently.
- Focus on whether the comment is accurate for the current diff, not whether the broader system could theoretically be improved.

## Step 1: Gather PR Comment Context
- Verify GitHub CLI auth with `gh auth status`.
- Identify the PR number or branch from user context.
- Fetch review threads, not just top-level issue comments.
- Prefer GraphQL so you can see thread IDs and resolution state.
- Collect:
  - thread ID
  - comment ID
  - file path and line
  - author
  - body
  - current commit SHA

Useful queries:

```bash
gh api graphql -f query='query { repository(owner:"<owner>", name:"<repo>") { pullRequest(number:<pr>) { reviewThreads(first:100) { nodes { id isResolved isOutdated comments(first:20) { nodes { id databaseId body path line author { login } } } } } } } }'
```

## Step 2: Analyze Correctness
- Read the referenced code and enough surrounding context to judge the claim.
- Distinguish:
  - real bug/regression introduced by the PR
  - valid local improvement worth taking
  - pre-existing repo-wide concern not introduced by the PR
  - inaccurate claim
  - technically true but not worth changing in this PR
- Be explicit about why.

## Step 3: If the Comment Is Accurate
- Implement the fix.
- Keep the fix scoped to the finding unless broader refactoring is clearly justified.
- Run targeted validation for the changed area.
- Commit and push before resolving the thread.
- Reply to the thread explaining:
  - that you agreed
  - what changed
  - what validation ran
- End every GitHub-facing reply with this exact line:

```text
Generated by {{RUNTIME_MODEL_ID}}
```

Example:

```text
Agreed. I fixed this in `<commit-sha>` by <brief fix summary> and reran `<validation>`.

Generated by {{RUNTIME_MODEL_ID}}
```

## Step 4: If the Comment Is Not Accurate or Not Worth Fixing
- Reply on the thread with a concise technical explanation.
- State whether the concern is:
  - not introduced by this PR
  - already handled elsewhere
  - inaccurate for the current code
  - intentionally out of scope for this change
- If relevant, mention the existing mechanism that contradicts the claim.
- End every GitHub-facing reply with this exact line:

```text
Generated by {{RUNTIME_MODEL_ID}}
```

Example:

```text
I do not agree this is a PR-specific finding. This code path already existed before this diff, and the new change only consumes the existing event stream without altering the trust boundary.

Generated by {{RUNTIME_MODEL_ID}}
```

## Step 5: Resolve Threads
- Resolve the thread only after one of these is true:
  - the fix is committed and pushed, and you replied
  - you replied with a clear dismissal rationale
- Use GraphQL to resolve:

```bash
gh api graphql -f query='mutation { resolveReviewThread(input:{threadId:"<thread-id>"}) { thread { id isResolved } } }'
```

- Do not leave an addressed thread unresolved.
- Do not resolve a thread silently without replying.

## Step 6: Final Report To The User
- Summarize:
  - which comments you agreed with
  - which comments you dismissed
  - commits pushed
  - validations run
  - whether all threads were resolved

## Rules
- Every GitHub-facing comment, reply, or review body must end with:
  - `Generated by {{RUNTIME_MODEL_ID}}`
- Resolve `{{RUNTIME_MODEL_ID}}` at runtime using the first non-empty environment variable in this order:
  - `OPENAI_MODEL`
  - `CODEX_MODEL`
  - `MODEL`
  - `LLM_MODEL`
- Use the resolved value as-is and do not guess a more specific model identifier.
- If no runtime model variable is available, use this exact fallback:
  - `Generated by Codex (runtime model id unavailable)`
- Recommended helper (use this output directly in GitHub replies):

```bash
resolve_runtime_model_id() {
  printf '%s' "${OPENAI_MODEL:-${CODEX_MODEL:-${MODEL:-${LLM_MODEL:-Codex (runtime model id unavailable)}}}}"
}
printf 'Generated by %s\n' "$(resolve_runtime_model_id)"
```

- Never post the literal placeholder `{{RUNTIME_MODEL_ID}}`.
- Do not miss this on dismissal replies. It applies equally to fixes and disagreements.
- Do not claim a finding is fixed until the commit is pushed.
- If the review comment exposes a real repo-wide issue outside the PR scope, say so clearly instead of pretending the PR should solve it.
